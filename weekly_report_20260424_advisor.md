## 本周推进方向

本周围绕两个核心问题推进。此前深耕过程中，系统迭代主要面临两个卡点：

1. **反思优化粗糙**：此前仅靠 fitness score 做粗粒度判断，无法精准定位哪个组件出了问题，导致迭代效率低
2. **组件权重固定**：奖励函数各分量的权重在生成时一次性定死，无法适配智能体不同训练阶段的学习需求

针对上述问题，本周完成了两项工作：一是将细粒度反馈闭环落地为可运行的迭代系统，二是结合近期文献规划了下一步的种群进化与层次化权重实验。

---

## 一、细粒度反馈闭环的落地

针对"反思优化粗糙"的问题，本周完成了基于结构化反馈的迭代闭环：

- **奖励分量轨迹追踪**：每步记录各分量的实时贡献值，输出 JSONL 轨迹日志。训练结束后可精确回答"agent 实际在优化哪个目标"——是前进速度不够，还是姿态惩罚过重压倒了其他信号
- **行为指标体系**：通过 `completion_rate`、`fall_rate`、`truncation_rate`、`mean_length` 等纯行为指标刻画 agent 的真实表现，完全不依赖 reward 数值
- **两阶段 LLM 迭代**：分析阶段（基于行为指标 + 分量轨迹做纯诊断，不写代码）→ 生成阶段（基于诊断结论定向改进）

这套闭环的思路借鉴了 R\* (ICML 2025) 的细粒度反馈诊断思想——不再用一个标量 fitness score 模糊地评价整个奖励函数，而是用训练过程的全程数据（训练曲线、分量贡献、轨迹片段）做结构化分析，让 LLM 能精准定位是哪个组件出了问题，而不是给出泛化的修改建议。当前实验正在运行中。

## 二、下一步实验规划：种群进化与层次化权重

### 方向一：种群进化搜索（借鉴 R\* 与 REvolve）

此前系统的迭代模式仍是单轮单候选的贪婪搜索，每轮只改进当前最优，搜索多样性不足。

**R\*** (Li et al., ICML 2025) 将奖励函数分解为结构演化 + 参数对齐两步，维护候选函数种群，通过模块级交叉和 LLM 反思做更充分的结构搜索。

**REvolve** (Hazra et al., ICLR 2025) 进一步将搜索显式改造为种群式进化框架，维护多个子种群，通过 mutation、crossover、selection、migration 等遗传操作持续演化，解决了 Eureka 贪婪搜索导致的过早收敛问题。

受这两篇工作启发，下一步实验将在现有框架基础上引入种群机制：每轮让 LLM 生成多个候选变体，通过快速验证筛选后保留优质个体，并探索模块级交叉组合。这与此前思考的"系统本质是在奖励空间中做结构化搜索，不应只盯着当前最优做局部修补"的方向一致。

### 方向二：层次化奖励组件的动态权重

当前奖励函数的各分量权重在生成时一次性定死，但智能体不同训练阶段的需求不同——前期应先掌握基础技能（保持平衡、不摔倒），后期再推进核心任务（走完全程）。

参考 Curriculum Learning 的渐进式目标调度思想，以及 R\* 将结构与参数解耦的设计思路，计划在后续实验中引入：将奖励组件划分为基础安全组件和任务目标组件，基础组件前期拉满权重作为硬约束，待完成率达到量化指标后再逐步降低、释放优化空间给任务目标组件。

### 方向三：模块化 agent 架构

借鉴 REvolve 中 population-based evolution 的多角色分工逻辑，以及此前阅读的 agent 框架研究中关于 skills 的思考，后续将对当前系统的 agent 结构进行模块化拆分——将奖励代码生成、训练执行、结果汇总、反思分析、候选筛选等环节拆分为可复用的 skill 模块，使系统既能保持长流程的自主性，又能在各环节独立改进和替换。
